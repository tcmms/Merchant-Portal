## I. КОГНИТИВНЫЙ ФРЕЙМВОРК (Mental Models)

1. **Composition-First Thinking:** Каждый компонент — изолированная, переиспользуемая единица. Предпочитай `children` и compound patterns пропсовому конфигурированию. Компонент должен делать одну вещь хорошо.
2. **Context Engineering:** Пиши код абсолютно идиоматично, чтобы архитектура была предсказуема для других инженеров и ИИ-агентов.
3. **AHA over DRY:** Приоритет принципу **AHA** (Avoid Hasty Abstractions). Дублирование кода дешевле неверной абстракции. Не создавай «компоненты-швейцарские ножи» с десятками пропсов.
4. **Accessibility by Default:** A11y — не опция, а стандарт. Семантический HTML, ARIA-атрибуты и keyboard navigation — это часть компонента, не afterthought.

## II. АРХИТЕКТУРА И СТРУКТУРА (Folder Structure)

Адаптированный **Atomic Design** под Vite + Storybook:

```
src/
├── components/
│   ├── ui/          # Atoms: примитивы без бизнес-логики (shadcn/ui)
│   └── shared/      # Molecules: комбинации атомов (FormField, Card и т.д.)
├── features/        # Organisms: фичи со своей логикой и состоянием
│   └── [feature]/
│       ├── components/   # Компоненты специфичные для этой фичи
│       ├── hooks/        # Хуки специфичные для этой фичи
│       └── index.ts      # Публичное API фичи
├── hooks/           # Глобальные переиспользуемые хуки
├── lib/             # Утилиты, helpers, конфигурация библиотек
├── types/           # Глобальные TypeScript типы и интерфейсы
└── stories/         # Storybook stories (если не рядом с компонентом)
```

**Правила:**
- Каждая папка компонента содержит `index.ts` — публичный экспорт.
- Stories (`.stories.tsx`) живут рядом с компонентом — `Button.stories.tsx` рядом с `Button.tsx`.
- Никаких circular imports — фичи не импортируют друг друга напрямую.

## III. ТЕХНИЧЕСКИЕ СТАНДАРТЫ (Senior-Grade Code)

### 1. React & Composition

- **SRP (Single Responsibility):** Один компонент — одна ответственность. Компонент, который слишком большой — признак что он должен быть разбит.
- **Composition:** Предпочитай паттерн **Compound Components** и `children` вместо prop drilling. Пример: `<Select>` + `<Select.Option>` вместо `<Select options={[...]} />`.
- **Controlled vs Uncontrolled:** Примитивы форм должны поддерживать оба режима. Используй паттерн `value` + `defaultValue`.
- **Render Props / Custom Hooks:** Логику UI (открыт/закрыт, активен/неактивен) выноси в хуки. Хуки не возвращают JSX.
- **Error Boundaries:** Каждая фича оборачивается в `ErrorBoundary`. Не давай ошибке одного виджета ронять всю страницу.

### 2. State Management

Используй минимально необходимый уровень стейта:

| Уровень | Инструмент | Когда |
|---------|-----------|-------|
| Локальный UI-стейт | `useState` | Открыт/закрыт, значение инпута |
| Сложный локальный стейт | `useReducer` | 3+ связанных поля, машина состояний |
| Серверный стейт | **TanStack Query** | Загрузка, кеш, синхронизация данных |
| Глобальный UI-стейт | **Zustand** | Тема, тостер, модалки, авторизация |
| Форм-стейт | **react-hook-form** | Любые формы |

**Запрещено:** Хранить серверные данные в `useState` + `useEffect`. Для этого есть TanStack Query.

### 3. Хуки

- Хуки атомарны, типизированы, не возвращают JSX.
- Называй по действию: `useModal`, `useTableSort`, `useDebounce`.
- `useEffect` — только для синхронизации с внешним миром (DOM, таймеры, подписки). Derived state вычисляй — не синхронизируй.
- Соблюдай правило: если `useEffect` синхронизирует стейт со стейтом — это ошибка архитектуры.

### 4. TypeScript Discipline

- **Zero `any` Policy:** Использование `any` запрещено. `unknown` + narrowing если тип неизвестен.
- **Discriminated Unions:** Моделируй UI-состояния через размеченные объединения:
  ```ts
  type RequestState<T> =
    | { status: 'idle' }
    | { status: 'loading' }
    | { status: 'success'; data: T }
    | { status: 'error'; error: string }
  ```
- **Schema-First:** Генерируй типы из Zod-схем: `z.infer<typeof schema>`.
- **Props Interfaces:** Всегда явно объявляй интерфейс пропсов компонента. Экспортируй его — разработчикам нужны типы.
- **Никаких `React.FC`:** Типизируй компонент через `({ prop }: Props) => JSX.Element`.

### 5. Forms

- **react-hook-form** — стандарт для любых форм.
- **Zod** — валидация схем на клиенте через `zodResolver`.
- Каждый инпут должен иметь: `label`, `error state`, `disabled state`, `required` индикатор.
- Форма без индикации загрузки и обработки ошибок — незаконченная форма.

### 6. Data Fetching

- **TanStack Query** — стандарт для серверного стейта.
- Каждый запрос должен обрабатывать три состояния: `isPending`, `isError`, `isSuccess`.
- Используй `queryKey` как массив с иерархией: `['users', userId, 'posts']`.
- Custom hook оборачивает каждый запрос: `useUser(id)`, `useProducts(filters)`.

### 7. Tailwind & Design System

- **8px Grid:** Предпочитай размеры кратные 4/8px. Для точных значений из Figma (`gap-[10px]`, `h-[54px]`) — используй Tailwind arbitrary values.
- **cva()** — для вариантов компонентов. **cn()** — для слияния классов.
- **Inline стили разрешены** для точных Figma-токенов: цвета (`rgba(0,0,0,0.45)`), box-shadow, borderColor. Tailwind-классы предпочтительны, но не в ущерб pixel-perfect точности.
- Анимации — через `transition-*` утилиты Tailwind или `framer-motion` для сложных случаев.

### 8. Accessibility (A11y)

- **Семантический HTML:** `<button>` для действий, `<a>` для навигации, `<nav>`, `<main>`, `<section>` по смыслу.
- **ARIA:** Добавляй `aria-label`, `aria-expanded`, `aria-controls`, `role` только когда семантического HTML недостаточно. Не дублируй то что уже выражено тегом.
- **Keyboard Navigation:** Все интерактивные элементы достижимы с клавиатуры. `Tab`, `Enter`, `Escape`, стрелки в меню — обязательно.
- **Focus Management:** При открытии модалки — фокус уходит в неё. При закрытии — возвращается на триггер.
- **Contrast:** Текст и UI-элементы соответствуют WCAG AA (4.5:1 для текста, 3:1 для UI).
- **Storybook a11y addon** — запускай на каждом компоненте. Нарушений быть не должно.

### 9. Performance

Применяй оптимизации только когда есть измеримая проблема:

| Инструмент | Когда использовать | Когда НЕ использовать |
|-----------|-------------------|----------------------|
| `React.memo` | Тяжёлый компонент в часто ре-рендерящемся родителе | По умолчанию на всё |
| `useMemo` | Дорогое вычисление (>1ms), зависит от пропсов | Примитивные вычисления |
| `useCallback` | Функция передаётся в `React.memo`-компонент | Просто "чтобы было" |
| `lazy` + `Suspense` | Тяжёлые фичи, которые не нужны при загрузке | Маленькие компоненты |

**Правило:** Преждевременная оптимизация — зло. Сначала измерь (React DevTools Profiler), потом оптимизируй.

## IV. DESIGN SYSTEM CONSTRAINTS (Storybook-First)

Перед любой UI-работой — обязательный первый шаг:

```
ALWAYS call the Storybook MCP server before writing any UI, frontend, or React code.
```

- **Storybook — единственный источник истины** по компонентам. Не изобретай то что уже есть.
- Используй компонент только через пропсы и варианты определённые в его story args.
- Если компонент есть в Storybook — используй его. Без обсуждений.
- Если компонента нет — сообщи пользователю и жди решения. Не создавай новый примитив молча.
- Не переопределяй стили существующих компонентов через inline styles или произвольные Tailwind-классы вне `cva()`.

## V. ТЕСТИРОВАНИЕ (Testing Strategy)

| Уровень | Инструмент | Что покрывать |
|---------|-----------|--------------|
| Хуки и утилиты | **Vitest** | Логика, edge cases, типы |
| Компоненты | **Storybook play()** | Интеракции, состояния, a11y |
| Интеграция | **Playwright** (опционально) | Критические user flows |

**Правила:**
- Хуки тестируются через `renderHook` из `@testing-library/react`.
- Компоненты тестируются через Storybook interaction tests — не дублируй в Vitest.
- Не мокируй то что можно использовать реально.
- Тест читается как документация: `it('shows error message when email is invalid')`.

## VI. ГИГИЕНА И ИМЕНОВАНИЕ (Conventions)

- **Events:** Внутренние обработчики — `handleFormSubmit`, пропсы — `onSubmit`.
- **Booleans:** Имена-вопросы: `isLoading`, `hasPermission`, `isOpen`, `isDisabled`.
- **Actions:** Сильные глаголы: `fetchUser`, `updateProduct`, `deleteComment`.
- **Files:** Компоненты — `PascalCase.tsx`, хуки/утилиты — `camelCase.ts`, константы — `SCREAMING_SNAKE_CASE`.
- **Exports:** Каждая папка имеет `index.ts`. Именованные экспорты везде — default export только для страниц/роутов.
- **Комментарии:** Только там где логика неочевидна. Комментарий объясняет **почему**, не **что**.

## VII. ПРОВЕРКА ПЕРЕД ВЫДАЧЕЙ (The "No-Go" List)

Перед тем как предложить решение, убедись что в нём **НЕТ**:

1. **Prop Drilling** более чем через 2 уровня.
2. **`useEffect`** для синхронизации стейта, который можно вычислить (Derived State).
3. **`any`** в TypeScript.
4. **Форм** без индикации загрузки и обработки ошибок.
5. **Интерактивных элементов** недостижимых с клавиатуры.
6. **`React.memo` / `useMemo` / `useCallback`** без измеримой причины.
7. **Magic numbers** в стилях — только дизайн-токены и 8px-сетка.
8. **Компонентов без Story** — если компонент создан, Story обязательна.

> **Исключение:** фича-специфичные компоненты в `features/` (например `StatusBadge`, `BulkActionBar`) могут не иметь Stories, если они не переиспользуются за пределами фичи.

### Протокол старта сессии

При старте каждой сессии читай **оба** источника контекста:

1. **`ai-context.md`** (корень проекта) — долгосрочная память: Learned, Patterns, Decisions, Recent Changes, Current Focus.
2. **`~/.claude/projects/-Users-temirlan-Merchant-Portal/memory/MEMORY.md`** — авто-память Claude Code: пользовательские предпочтения, фидбэк, ссылки на внешние ресурсы.

### Минимальный контекст для старта задачи

Прежде чем приступить к любой задаче, убедись что есть:

- **Что:** чёткое описание (не одно слово)
- **Где:** затронутые компоненты или фича
- **Почему:** пользовательский сценарий или дизайн-решение
- **Ограничения:** что нельзя трогать

Если любой из этих пунктов отсутствует — задай один уточняющий вопрос. Не начинай писать код на предположениях.

### Персистентная память — два файла, разные роли

| Файл | Что хранить | Когда обновлять |
|------|-------------|-----------------|
| **`ai-context.md`** | Learned (что сработало/провалилось), Patterns (proven/experimental/avoid), Decisions Log, Recent Changes, Current Focus | После каждой сессии и при завершении задач. Это **главный** файл для технического опыта. |
| **`memory/MEMORY.md`** + файлы | Пользовательские предпочтения (user), фидбэк по процессу (feedback), статус проекта (project), ссылки (reference) | Когда узнаёшь что-то о пользователе, его предпочтениях, или внешних ресурсах — то что НЕ выводится из кода. |

**Правило:** технические решения, паттерны, ошибки → `ai-context.md`. Всё остальное → `memory/`.

## IX. АРХИТЕКТУРА ПРОЦЕССА (The Engine)

### Последовательная агентность

Работай как конвейер ролей: `[Planner] → [Executor] → [QA]`.

- **Planner:** декомпозирует задачу, предлагает архитектуру, ждёт подтверждения.
- **Executor:** пишет код строго по согласованному плану.
- **QA:** проверяет по No-Go List, запускает Storybook, проверяет a11y.

Не смешивай фазы. Не переходи к Executor без подтверждения плана пользователем.

### Chain of Thought

Используй блоки `<thinking>` для декомпозиции. Объясняй архитектурные выборы до написания кода.

## XI. БЕЗОПАСНОСТЬ И ЭСКАЛАЦИЯ (The Safety)

### Circuit Breakers

Если совершаешь одну и ту же ошибку трижды или застрял в типах/линтере на две итерации — остановись. Опиши проблему и жди вмешательства человека.

### Эскалация

Точность важнее скорости. При любом архитектурном конфликте — немедленно эскалируй с вариантами решения на выбор.

## XII. ОБУЧЕНИЕ И САМОСОВЕРШЕНСТВОВАНИЕ (The Fine-Tuning)

### Накопление опыта

После каждой сессии обновляй `ai-context.md` → `## Learned`:

- **Что сработало:** паттерн или решение и почему.
- **Что провалилось:** что пошло не так, как избежать.
- **Решения принятые намеренно:** архитектурные выборы которые нельзя пересматривать молча.

### Градация паттернов

Веди реестр в `ai-context.md` → `## Patterns`:

- ✅ **Proven** — использован 2+ раза, работает стабильно. Применяй по умолчанию.
- 🧪 **Experimental** — использован однажды, результат неоднозначен. Сообщай при применении.
- ❌ **Avoid** — провалился или создал регрессию. Не применяй без явного запроса.

### Аудит качества

Запускается командой `/audit`. Оценка по трём осям:

- **Скорость:** задачи решаются за меньше итераций?
- **Качество:** меньше обращений к Circuit Breaker?
- **Паттерны:** соотношение Proven к Experimental растёт?
